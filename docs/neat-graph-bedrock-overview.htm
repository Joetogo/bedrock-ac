01 · What this is

One gateway, three telemetry sources, plain-English queries

Amazon Bedrock drives a tool-use loop against a single AgentCore Gateway, which is the MCP server. Each upstream is a Lambda exposed as MCP tools; the Gateway injects each service's credential server-side, so the model never sees an API key.

The goal this session: prove all three integrations answer real questions through the deployed gateway — then design a hosted agent so those questions can be asked without running local scripts.

  NL query
    │  (Bedrock Converse tool-use loop)
    ▼
  AgentCore Gateway  ── MCP endpoint · Cognito JWT inbound auth
    ├── neat          → Lambda → Neat Pulse / Sense API   (Bearer · Secrets Mgr)
    ├── graph         → Lambda → MS Graph callRecords     (client credentials)
    ├── thousandeyes  → Lambda → ThousandEyes v7 REST     (OAuth Bearer)
    └── correlate     → Lambda → joins room ↔ call quality

Security property

Every tool is read-only by construction (GET requests only). Upstream credentials live in AWS Secrets Manager and are injected by the Lambda at call time — the Neat API key, the Graph secret, and the ThousandEyes token never enter the model context or .env.

02 · Incident & fix log

What was broken, and what fixed it

Every layer of the plumbing was healthy — Cognito auth, the gateway, tool discovery, the Bedrock loop. The failures were all downstream: a placeholder secret, a wrong API path, a mis-packaged Lambda layer, and a ThousandEyes auth mismatch.

  1. Neat secret held REPLACE_ME

    Real credentials existed only in an ap-southeast-2 secret under mismatched key names; the live us-east-1 stack read placeholders, so orgs/REPLACE_ME/rooms correctly 404'd.

    FIX → copied real org_id / api_key into the us-east-1 secret

  2. Neat sensor path was plural

    Code called /rooms/{id}/sensors; the Neat Pulse API path is singular /sensor. The API's own hint ("no matching operation") confirmed it.

    FIX → handler.py + test_integrations.py → /sensor

  3. Lambdas crashed at import: No module named '_shared'

    The shared layer was packaged as python/python/_shared/ — a double nesting. build_layer.sh added a python/ level and SAM's layer builder added its own, so the module was never importable. The neat/graph/correlate tools had never actually run.

    FIX → stage _shared at the layer root; rebuild → single python/_shared/; redeploy

  4. ThousandEyes 401 → 403 → 200

    The supplied 32-char credential was a legacy v6 User API Token (Basic auth, needs a username) — it returned 200 only on v6 status.json and 403 on every real resource, on both API versions. The v7 API requires an OAuth Bearer Token. Once a proper OAuth token was generated, v7 returned 200 with 57 real tests.

    FIX → OAuth Bearer Token in the thousandeyes secret; te_get reverted to clean Bearer

  5. Region split & console polish

    The live stack is us-east-1; orphan secrets lingered in ap-southeast-2 (no stack there). The Makefile default region and an emoji-unsafe console print were corrected.

    FIX → Makefile default → us-east-1; ask.py forces UTF-8 stdout

03 · ThousandEyes auth — the decisive detail

v6 User API Token vs v7 OAuth Bearer

The docs settled a long detour. The official ThousandEyes MCP confirms v7 auth is Bearer, no username — the "requires a username" symptom was the v6 model, a dead end for this integration.

Auth methodEndpointResult
Bearer <v6 token>v7 /tests401 invalid_token
Basic user:tokenv7 /tests403 forbidden
Basic user:tokenv6 /status.json200 (no perms needed)
Basic user:tokenv6 /tests.json403 forbidden
Bearer <OAuth token>v7 /tests200 · 57 tests

Takeaway: a rebuild onto v6 would not have helped — every data endpoint 403'd on both versions with the old token. The fix was the token type, not the API version. v7 is also the current, non-deprecated surface.

05 · Next phase · approved design

AgentCore Runtime harness (Strands)

Move the ask.py Converse loop off the laptop and host it as an AgentCore Runtime agent built with the Strands Agents SDK, so the deployment answers plain-English questions via the console / CLI — with AgentCore Memory for multi-turn recall.

  Operator (CLI: agentcore invoke) — {prompt, sessionId}
    ▼
  AgentCore Runtime  (containerized Strands agent)
    ├── Strands Agent  → Bedrock (auto-selected Claude profile)
    ├── AgentCore Memory  ← recalls prior turns by sessionId
    └── MCPClient ──► existing Gateway MCP  (Cognito JWT)
                        └── neat_* / graph_* / te_* / correlate_*  (read-only)
DecisionChoice
Query surfaceBedrock / AgentCore console + CLI no UI to build
MemoryMulti-turn via AgentCore Memory, keyed by sessionId
FrameworkStrands Agents SDK on AgentCore Runtime

Read-onlypreserved by construction — only gateway GET tools reachable
